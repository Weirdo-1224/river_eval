# 统一长程视频交互测评框架开发目标

## 1. 项目定位

本项目旨在构建一个可长期复用的长程视频交互测评框架。框架初期以 RIVER 作为标准任务来源，完成通用长程视频理解任务的复现与评测；后续逐步过渡到实验室自建的电力作业安全监督场景，并用于评估本地多模态模型及其记忆机制改进效果。

本框架的核心目标不是只复现某一个 benchmark，而是形成一套稳定的测评基础设施。通过解耦 Dataset、Task、Model、Video、Memory、Evaluator 和 Runner，保证后续更换任务、数据集、模型或评价体系时，不需要重写整体框架。

因此，本项目的长期定位是：

```text
从 RIVER 通用长程视频交互测评出发，逐步扩展到电力作业安全监督及更多长程视频场景的统一测评框架。
```

## 2. 总体路线

项目按照四个阶段推进。

### 第一阶段：基于 RIVER 完成框架解耦

第一阶段以 RIVER 为任务协议来源，先完成通用测评框架的搭建。所有任务先注册为 RIVER task，包括：

```text
river_retro_memory
river_live_perception
river_pro_response_instant
river_pro_response_streaming
```

这一阶段的重点是把测评流程拆清楚：

```text
Dataset
Task
Model
Video Policy
Prompt Protocol
Scoring
Result Writer
Trace Logger
Runner
```

Runner 只负责读取配置和组织流程，不再写死具体任务、模型、抽帧方式或评分方式。每次实验都应通过配置文件明确声明：

```text
使用哪个数据集
使用哪个任务
使用哪个模型
使用哪种抽帧策略
使用哪种 prompt 协议
使用哪种评分方式
```

第一阶段需要优先严格复现 RIVER 的任务逻辑，尤其是 Retro-Memory 的标注读取、question_time 使用、long memory 与 short clip 构造、prompt、选项解析和准确率计算。其他任务可以先完成接口注册，再逐步补齐严格实现。

### 第二阶段：接入本地开源多模态模型

第二阶段在框架已经解耦的基础上，接入本地开源多模态模型，例如 Qwen-VL、InternVL、MiniCPM-V、GLM-V 等。

这一阶段不改变测评任务本身，仍然主要使用 RIVER task 作为统一测试集。重点是验证不同模型能否通过统一 Model Adapter 接入同一套测评流程。

本阶段目标是形成稳定的模型横向评测能力，回答：

```text
不同多模态模型在长程视频理解、历史记忆和主动响应任务上的表现如何。
哪个模型更适合作为后续电力作业场景的基础模型。
```

### 第三阶段：构建自定义任务与评价体系

第三阶段开始从 RIVER 任务过渡到实验室自建任务。此时不改变整体框架，只是在现有解耦结构中新增自定义 Dataset、Task 和 Evaluator。

面向电力作业场景，可以逐步构建以下任务：

```text
作业步骤识别
前置条件检查
流程顺序判断
违规行为检测
主动风险预警
规程符合性判断
长程作业状态追踪
```

同时，需要建立适合该场景的评价体系，例如：

```text
违规识别准确率
前置步骤缺失识别率
流程顺序错误识别率
风险等级判断准确率
报警及时性
误报率
漏报率
重复报警率
规程链条一致性
解释正确性
```

这一阶段的核心是将 RIVER 的通用长程视频交互测评能力，迁移到电力作业安全监督任务中。

### 第四阶段：更新本地模型记忆模块并完成任务验证

第四阶段进入模型改进与研究验证阶段。此时 benchmark 已经具备稳定的数据、任务、模型、视频、评分和 trace 记录能力，可以作为核心测评工具，用于验证本地模型记忆机制的改进效果。

需要比较不同记忆机制，例如：

```text
NoMemory
SlidingWindowMemory
TokenMemory
EventMemory
GraphMemory
RuleStateMemory
```

对于电力作业任务，重点是规程状态记忆，即模型不仅要记住过去画面，还要维护作业流程状态、前置条件完成情况、安全检查状态、风险触发记录等信息。

本阶段目标是验证：

```text
改进后的记忆模块是否提升长程理解能力
是否减少遗忘
是否提升违规检测准确率
是否降低误报和漏报
是否提高主动风险响应及时性
```

## 3. 框架不变性原则

整个项目虽然会从 RIVER 过渡到电力作业场景，但 benchmark 主框架不应反复推倒重写。框架应保持稳定，变化的只是可插拔组件。

新增数据集时，只新增 Dataset Adapter。

新增任务时，只新增 Task。

新增模型时，只新增 Model Adapter。

新增抽帧方式时，只新增 Video Policy。

新增评价指标时，只新增 Evaluator。

新增记忆机制时，只新增 Memory Module。

Runner 始终保持配置驱动，不随具体任务扩展而变复杂。

## 4. 核心框架结构

最终框架应保持如下结构：

```text
Dataset
    负责读取 RIVER 或自建数据，并转换为统一样本格式

Task
    负责定义任务协议、prompt、可见范围、输出解析方式

Video Policy
    负责抽帧、窗口划分、long memory 与 short clip 构造

Model Adapter
    负责统一调用 API 模型和本地模型

Memory
    负责维护历史信息、事件状态和规程状态

Evaluator
    负责计算准确率、响应延迟、误报率、漏报率等指标

Storage / Trace
    负责保存结果、过程轨迹、缓存和成本统计

Runner
    负责读取配置并组织完整评测流程
```

## 5. 配置驱动方式

每次实验都应由配置文件控制，例如：

```yaml
dataset:
  name: river_retro_memory

task:
  name: river_retro_memory

model:
  name: qwen_api

video:
  frame_policy: river_long_short

prompt:
  style: river_longshort

evaluation:
  scoring: river_mcq_accuracy

storage:
  save_trace: true
  use_cache: true
```

后续切换到电力作业任务时，只需要替换配置中的 dataset、task 和 evaluation，而不需要改主流程。

## 6. 最终目标

本项目最终形成一个统一的长程视频交互测评框架。它首先用于复现和扩展 RIVER，随后用于评估本地开源多模态模型，再进一步承载电力作业安全监督任务，最后用于验证本地模型记忆机制改进效果。

整体路线可以概括为：

```text
RIVER 通用测评
        ↓
统一解耦测评框架
        ↓
本地开源模型接入
        ↓
自建电力作业任务与评价体系
        ↓
本地模型记忆机制改进与验证
        ↓
扩展到更多长程视频交互场景
```

框架的核心价值在于：通过解耦设计，使同一套 benchmark 能够从通用视频交互测评自然扩展到电力作业安全监督，并继续扩展到更多需要长程理解、实时交互和主动响应能力的复杂视频场景。
